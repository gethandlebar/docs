## PII & Financial Data Boundaries

**Problem**: Agents don't intuitively understand what constitutes Personally Identifiable Information (PII) or sensitive financial data under regulations like GDPR, GLBA, and PCI-DSS.

**Handlebar's approach**:
- Tag tools and data sources as PII-touching or financial-data-sensitive
- Define role-based access rules
- Block tool calls that would expose PII or financial data to unauthorised contexts

**Example rule**: 
```
IF tool.touches_pii == TRUE
AND requester.role NOT IN ["compliance_officer", "authorized_analyst"]
THEN action = BLOCK
```

## External API & Third-Party Controls

**Problem**: Agents might send customer data to external APIs, analytics platforms, or third-party services without proper data processing agreements.

**Handlebar's approach**:
- Flag external tool calls
- Require approval or block by default for data leaving organizational boundaries
- Log all external disclosures for vendor management and audit compliance

**Example rule**:
```
IF tool.destination == "external"
AND data.contains_financial_info == TRUE
AND vendor.dpa_status != "approved"
THEN action = BLOCK
```

## Customer Data Integrity

**Problem**: Agents might mix up customer contexts mid-session (e.g., start analysing Customer A's portfolio, switch to Customer B's transaction data).

**Handlebar's approach**:
- Track active customer context per run
- Block tool calls that reference the wrong customer
- Log context switches for audit and surveillance

**Example rule**:
```
IF run.active_customer_id != tool_call.customer_id
THEN action = BLOCK
```

## Trading & Transaction Constraints

**Problem**: AI agents making or recommending trades must operate within pre-approved parameters and comply with suitability requirements, position limits, and firm policies.

**Handlebar's approach**:
- Tag agents by authorisation level and trading mandate
- Define tool access rules per agent type and customer profile
- Enforce transaction limits and approval workflows at runtime

**Example rule**:
```
IF tool == "execute_trade"
AND trade.amount > agent.max_trade_limit
THEN action = REQUIRE_APPROVAL
```

**Example rule**:
```
IF tool == "recommend_investment"
AND customer.risk_profile == "conservative"
AND product.risk_rating > 3
THEN action = BLOCK
```

## Model Risk & Bias Controls

**Problem**: Different models have different risk profiles and potential biases. Financial decisions require explainable, fair, and auditable AI.

**Handlebar's approach**:
- Tag agents by model type and intended use
- Restrict high-risk tools to validated, audited models
- Block or require approval for sensitive decisions from unvalidated models

**Example rule**:
```
IF agent.model_validation_status != "approved"
AND tool IN ["credit_decision", "loan_approval", "risk_assessment"]
THEN action = BLOCK
```

## Market Abuse & Surveillance

**Problem**: Agents must not engage in or facilitate market manipulation, insider trading, or other prohibited activities.

**Handlebar's approach**:
- Monitor for patterns consistent with market abuse
- Block tool calls that could constitute front-running or wash trading
- Escalate suspicious activity to compliance teams
- Maintain immutable audit trail for regulatory examination

**Example rule**:
```
IF tool == "execute_trade"
AND run.recent_trades_same_security > 5
AND run.time_window < "10_minutes"
THEN action = ESCALATE
reason = "Potential wash trading pattern detected"
```

## Know Your Customer (KYC) Enforcement

**Problem**: Agents interacting with customers must verify identity and comply with KYC/AML requirements before providing services or executing transactions.

**Handlebar's approach**:
- Verify KYC status before allowing account access or transactions
- Block operations on accounts with incomplete or expired KYC
- Require approval for high-risk customer interactions

**Example rule**:
```
IF customer.kyc_status IN ["expired", "incomplete", "rejected"]
AND tool IN ["open_account", "execute_trade", "process_withdrawal"]
THEN action = BLOCK
```

## Regulatory Reporting & Recordkeeping

**Problem**: Financial firms must maintain detailed records of decisions, recommendations, and transactions for regulatory review (SEC, FINRA, FCA, etc.).

**Handlebar's approach**:
- Immutable audit logs for every tool call and decision
- Full replay capability using versioned rules
- Export to regulatory reporting formats (MIFID II, Dodd-Frank, etc.)
- Timestamping compliant with securities regulations

This is where determinism pays off. You can show regulators exactly why a decision was made, prove that your agent complied with suitability requirements, and reconstruct any customer interaction.

## Fairness & Anti-Discrimination

**Problem**: AI agents making credit, lending, or insurance decisions must comply with fair lending laws and avoid discriminatory outcomes.

**Handlebar's approach**:
- Block access to protected attributes (race, gender, religion) in decision-making tools
- Require human review for decisions that could have disparate impact
- Monitor and log all credit and underwriting decisions for bias testing

**Example rule**:
```
IF tool == "credit_underwriting"
AND tool_call.inputs CONTAINS_ANY ["race", "ethnicity", "gender", "religion"]
THEN action = BLOCK
reason = "Protected attribute detected in credit decision"
```
