## PHI Boundaries

**Problem**: Agents don't intuitively understand the definition of Protected Health Information.

**Handlebar's approach**:
- Tag tools and data sources as PHI-touching
- Define role-based access rules
- Block tool calls that would expose PHI to unauthorised contexts

**Example rule**: 
```
IF tool.touches_phi == TRUE
AND requester.role NOT IN ["clinician", "authorised_admin"]
THEN action = BLOCK
```

## External Disclosure Controls

**Problem**: Agents might send PHI to external APIs, logging services, or third-party tools.

**Handlebar's approach**:
- Flag external tool calls
- Require approval or block by default
- Log all external disclosures for compliance

**Example rule**:
```
IF tool.destination == "external"
AND data.contains_phi == TRUE
THEN action = REQUIRE_APPROVAL
```

## Patient Data Integrity

**Problem**: Agents might mix up patient contexts mid-conversation (e.g., start discussing Patient A, switch to Patient B's data).

**Handlebar's approach**:
- Track active patient context per run
- Block tool calls that reference the wrong patient
- Log context switches for audit

**Example rule**:
```
IF run.active_patient_id != tool_call.patient_id
THEN action = BLOCK
```

## Model Usage Constraints

**Problem**: Different models have different risk profiles. You might allow GPT-4 for clinical summaries but not for prescriptions.

**Handlebar's approach**:
- Tag agents by model type
- Define tool access rules per model
- Enforce at runtime

**Example rule**:
```
IF agent.model == "gpt-3.5"
AND tool == "generate_prescription"
THEN action = BLOCK
```

---

## Incident Reconstruction

**Problem**: When something goes wrong, you need to know exactly what the agent did, why, and whether it violated policy.

**Handlebar's approach**:
- Immutable audit logs for every tool call
- Full replay capability using versioned rules
- Export to compliance reporting formats

This is where determinism pays off. You can show an auditor exactly why a decision was made.
